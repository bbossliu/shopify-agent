# Shopify 商品内容 Agent · 项目计划

> **一句话**：当 Shopify 里新增/更新商品时，webhook 触发一个 AI agent，读取商品描述与评论，**生成与店铺现有体系一致、且不瞎编**的标签、分类和 SEO 文案，经置信度分级 + 人工审核后写回。
>
> **简历一句话**：Built a production, webhook-driven AI agent that auto-enriches Shopify product catalogs (tags / categorization / SEO copy) — RAG-grounded for catalog consistency, with faithfulness evaluation and confidence-gated human-in-the-loop review.
>
> 建议 repo 名：`shopify-content-agent`（可改）

## 贯穿全程的原则

- [ ] 承重原语亲手写一次（裸 loop + 三个 aha），其余交给 AI、但我审得了。
- [ ] **评估和审查永远自己扛**，不外包。
- [ ] 让 AI 当老师：要它解释、追问、改它的代码、故意弄坏看我懂不懂。
- [ ] 及格线：跑之前能不能预测这段代码干什么？错了能不能看出来？
- [ ] **agent 核心与 Shopify 解耦**：核心只跟一个接口打交道，Shopify 是后加的一层。
- [ ] 接口干净自包含——随时自问：这块将来能不能原样塞进一个 subagent 跑？

## 阶段一 · 亲手筑地基（在 mock 数据上）

不碰 Shopify，不碰 LangGraph。用原始 Messages API 手写。

- [ ] **裸 loop**：问模型 → 解析 tool_use → 执行 → 结果塞回 messages → 再问。
  - aha：短期记忆就是 `messages` 数组本身。
- [ ] **mock 数据**：用普通 dict 造几个假商品（标题、描述、若干条评论）。
- [ ] **手写工具**（先用 mock 实现，返回假数据 / 只打印）：
  - `get_product(id)` / `get_reviews(id)` / `propose_tags` / `propose_category` / `write_seo`
  - 重点：返回内容精简、出错反馈可读。这步决定"效果好不好"。
- [ ] **撞上下文墙**：塞进超长描述 + 几十条评论，看 messages 逼近上限。
- [ ] **手写压缩**：评论太多时，先摘要再喂。aha：压缩 = 总结 + 替换。
- [ ] **手写长期记忆的最小版**：把"店铺已有标签体系/分类表"放进一个外部存储 + 一个检索工具读。
  - aha：agent 不"拥有"标签体系，它是去"检索"的——这颗种子直接长成阶段四的 RAG。

**产出物**：一个不依赖框架的最小 agent，能对 mock 商品产出标签/分类/SEO。

---

## 阶段二 · 补齐 agent 能力（AI 写实现，我来审）

- [ ] **按需加载**：别把所有评论一次塞进去，让 agent 需要时再抽样/拉取。
- [ ] **结构化输出 + 校验 + 护栏**：
  - 标签必须来自允许的词表；分类必须在已有分类表里；SEO 标题 ≤ 限定字数。
  - schema 校验，违规就拒绝/重试。
- [ ] **置信度 + 人在回路**：agent 对每条提议给置信度；低置信打标进人工审核（不直接写回线上）。
- [ ] **忠实度护栏**：禁止生成商品源数据里没有的卖点/规格。

---

## 阶段三 · 评估（自己扛 · 本项目最大亮点）

- [ ] **建评估集**：N 个商品 + 标准答案（gold 标签/分类）+ 参考文案要点。
- [ ] **硬指标**：标签的精确率/召回率、分类准确率，对着 gold 算。
- [ ] **LLM-as-judge**：SEO 文案质量（品牌话术契合、长度、关键词覆盖）。
- [ ] **忠实度评估**：判定有没有瞎编源数据里没有的内容（最值钱的一项）。
- [ ] **回归**：改了 prompt / 检索后，能说出"这次让标签召回 +X%、瞎编率 -Y%"。

**产出物**：一套评估 harness，能量化每次改动是变好还是变差。

---

## 阶段四 · 检索 / RAG（让输出"一致"而非"瞎编"）

> 这是本项目 RAG 的承重点：不是从零造词，而是**贴着店铺已有的体系来生成**。

- [ ] 把已有商品目录 embedding 入库。
- [ ] 检索三类上下文喂给生成：
  - (a) 相似商品 + 它们的标签 → 约束标签词表，避免每个商品各发明一套；
  - (b) 已有的优质 listing 作为 few-shot → 对齐格式与品牌话术；
  - (c)（可选）SEO 关键词。
- [ ] **单独评估检索这一步**：捞到的相似商品对不对（召回/精度）。
- [ ] **对照实验**：有 RAG vs 无 RAG，用阶段三的指标证明"一致性上升、瞎编下降"。
  - 这组对照数字是 README 里最漂亮的故事。

**产出物**：接真实（或较大的 mock）目录，并能量化检索好不好。

## 阶段五 · 工程化 / 上生产（webhook 的主场）

- [ ] **真实对接 Shopify**：开发店、OAuth、注册 `products/create` + `products/update` webhook。
  - ⚠️ 评论源（Judge.me / metafields 等）作为可替换的一层后接，核心不依赖它。
- [ ] **webhook 加固**：HMAC 签名校验、幂等（按 webhook id / 商品版本去重）、**异步队列**（先入队立刻回 200，别阻塞）、重试。
- [ ] **可观测性**：每次运行记录——调了哪些工具、检索到什么、置信度、最终决定。
- [ ] **成本 / 延迟**：按商品算 token 成本；品牌话术/系统上下文用 prompt caching。
- [ ] **写回**：审核通过的提议，通过 Admin API 写回标签、SEO 字段、分类。
- [ ] 本地开发：内网穿透（tunnel）接收 webhook。

**产出物**：流式、可观测、能算成本、出错优雅兜底的生产形态。

## 附：单 vs 多 agent（本项目的判断）

本项目是**单 agent**：一个 webhook = 一个商品 = 一个独立任务。
不需要多 agent，因为子任务（标签/分类/文案）彼此耦合、共享同一份商品上下文，拆开只会增加协调成本。
→ 但保持 loop 干净自包含，将来若要"批量重处理整个目录"，可以平滑 fan-out 成 orchestrator-worker。
